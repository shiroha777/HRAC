# 为什么 HRAC 在攻击者权重已经很低时仍无法很好更新？

从 `hrac-log-bit_flipping.txt` 可以看到，攻击者 (client 7,8,9) 的 weight 经常被压到 0.002~0.01，但 accuracy 仍会掉到 0.1~0.15，甚至出现 "Clients with ν > 1: 10/10" 后崩溃。原因主要有三点。

## 1. 方向一致 + 每轮残留

聚合是 **加权和**：`g_t = Σ w_i * update_i`。即使每个攻击者权重只有 0.003，3 个攻击者 **方向一致** 时，在攻击方向上的总贡献是：

- `3 × 0.003 × ||update_attacker|| ≈ 0.009 × (攻击向量)`
- 单轮看占比小，但 **每一轮** 都往同一错误方向推一点，多轮累积会明显拉偏模型。

诚实节点方向更分散，在攻击方向上的分量会被部分抵消；攻击者则每轮都叠加。

## 2. “ν>1: 10/10” 时诚实节点也被惩罚

当前惩罚是 `excess = max(ν-1, ν-2×median) + ...`。当某一轮 **所有** 客户 ν>1（例如 iter 600：honest ν≈2~4，attacker ν≈30），则：

- 诚实客户也有 `excess_above_1 = ν-1`（例如 1~3），权重会从 0.1 降到约 0.09；
- 攻击者 excess 更大，权重约 0.01；
- 归一化后，诚实总权重约 0.63，攻击者约 0.03，**攻击者占比约 5%**。

也就是说，在 “全员 ν>1” 的轮次里，诚实节点一起被罚，攻击者的 **相对份额** 反而会上升，单轮 5% 的错误方向每轮叠加，仍会拖垮准确率。

## 3. 之前 outlier 项在 median 被拉高时失效

旧公式里有一项：`excess_outlier = (ν - 3×median)^+`。当 “ν>1: 10/10” 时，median(ν) 被攻击者拉高（例如 ≈11），则：

- `3×median ≈ 33`，攻击者 ν≈30 时 `excess_outlier = 0`，**outlier 惩罚不起作用**。

因此需要改成 **相对 median 的超出量**：`(ν - median)^+`。这样：

- 诚实 ν≈2、攻击者 ν≈30、median≈10 时，只有攻击者有 `(30-10)=20` 的额外 excess；
- 即使 median 被拉高，只要攻击者 ν 明显高于 median，就会继续被额外惩罚。

## 本次改动

1. **Outlier 惩罚改为 (ν - median)^+**  
   - 使用 `excess_above_median = max(0, ν - median(ν))`，再乘以 `nu_penalty_outlier_scale`（默认 2.0）加进 `nu_excess`。  
   - 这样 “ν 不仅大于 1，而且远超其他 client” 的会被额外加大惩罚。

2. **默认加大惩罚**  
   - `nu_penalty_outlier_scale` 默认从 1.0 改为 2.0，使 “超出 median 的部分” 惩罚更重。

3. **公式形式**  
   - `nu_excess = max(ν-1, ν-2×median) + nu_penalty_outlier_scale * (ν - median)^+`  
   - 再 `raw_weights = 1 / (1 + alpha * nu_excess)`，归一化后得到权重。

重新跑 bit_flipping 实验时，在 “ν>1: 10/10” 的轮次里，攻击者应会被压得更低，诚实节点相对份额提高，有利于稳定更新和准确率。

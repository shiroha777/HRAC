# 衰减参数与 Recall 表现的关系分析

## 发现的衰减参数

代码中存在以下随时间衰减的参数：

### 1. **tau_t_ema** (阈值 EMA 平滑) ⚠️⚠️⚠️
- **位置**: `ByrdLab/aggregation.py:1091-1097`
- **衰减系数**: `tau_ema_beta = 0.95`
- **计算方式**: 
  ```python
  tau_t_ema = tau_ema_beta * self.tau_t_ema + (1.0 - tau_ema_beta) * tau_t_raw.item()
  ```
- **含义**: 使用指数移动平均 (EMA) 平滑阈值，新值权重仅 5%，历史值权重 95%

### 2. **rho** (历史对齐衰减) ⚠️⚠️
- **默认值**: `0.9`
- **使用位置**:
  - `client_history_c` (历史对齐累积): `cur_T = self.rho * prev_T + update_weight * align_centered[i]`
  - `client_history_r` (历史范数): `self.client_history_r[cid] = self.rho * self.client_history_r[cid] + (1.0 - self.rho) * msg_norms_clipped[i]`
  - `client_sketch_history` (sketch 历史): `cur_sketch_ema = rho_sketch * prev_sketch_ema + (1.0 - rho_sketch) * sketch_sign[i]`
- **含义**: 历史信息权重 90%，当前信息权重 10%

### 3. **beta** (服务器动量衰减)
- **默认值**: `0.9`
- **使用位置**: `self.server_momentum = self.beta * self.server_momentum + (1.0 - self.beta) * aggregated_update`
- **含义**: 服务器动量使用 EMA 平滑

## 关键发现

### 问题 1: **tau_t EMA 累积效应导致阈值"锁定"** 🔴🔴🔴

**现象**:
- 失败案例的 `tau_t` 均值: **-9.55** (非常负)
- 成功案例的 `tau_t` 均值: **-0.05** (接近 0)
- 失败案例的 `tau_t` 最小值: **-408.07** (极端负值)
- 失败案例的 `tau_t` 标准差: **42.45** (方差极大)

**原因**:
1. **EMA 累积效应**: 当早期迭代出现极端负的信任分数时，`tau_t_ema` 会累积这些负值
2. **恢复缓慢**: 由于 `beta=0.95`，新值权重仅 5%，需要约 20 轮才能显著改变阈值
3. **"锁定"现象**: 一旦 `tau_t_ema` 变得非常负，即使后续信任分数改善，阈值也很难恢复

**时间趋势分析**:
- **早期阶段** (0-5000 迭代):
  - 失败案例 `tau_t` 均值: **-19.49**
  - 成功案例 `tau_t` 均值: **-0.05**
  - 阶段内趋势: -30.80 -> -0.05 (变化: +30.74，说明成功案例能快速恢复)
  
- **中期阶段** (5000-10000 迭代):
  - 失败案例 `tau_t` 均值: **-10.78**
  - 成功案例 `tau_t` 均值: **-0.06**
  - 阶段内趋势: -21.90 -> -0.05 (变化: +21.85)
  
- **后期阶段** (10000+ 迭代):
  - 失败案例 `tau_t` 均值: **-0.03** (已恢复)
  - 成功案例 `tau_t` 均值: **-0.04**
  - 阶段内趋势: -0.03 -> -0.04 (变化: -0.004，基本稳定)

**结论**: 早期阶段的极端负值会通过 EMA 累积，导致阈值被"锁定"在负值，影响后续检测。

### 问题 2: **rho 过高导致历史信息主导** 🟠🟠

**现象**:
- `rho=0.9` 意味着历史信息权重 90%，当前信息权重仅 10%
- 如果攻击者早期获得正对齐分数，历史会持续影响后续判断

**影响**:
1. **历史对齐累积**: `client_history_c` 使用 `rho=0.9` 累积历史对齐
   - 如果攻击者早期获得正对齐，历史会持续"掩盖"当前负对齐
   - 即使当前轮对齐为负，历史仍可能主导权重计算

2. **历史范数累积**: `client_history_r` 使用 `rho=0.9` 累积历史范数
   - 如果早期范数异常，会影响后续尺度偏差计算

3. **Sketch 历史累积**: `client_sketch_history` 使用 `rho=0.9` 累积 sketch 历史
   - 影响自一致性计算

## 衰减参数与 Recall 失败的关系

### 直接关系

1. **tau_t EMA 累积** → **阈值失效** → **Recall 失败**
   - 早期极端负值 → EMA 累积 → 阈值被锁定 → 无法区分攻击者 → Recall=0%

2. **rho 过高** → **历史主导** → **当前信息被掩盖** → **Recall 失败**
   - 早期正对齐 → 历史累积 → 即使当前负对齐，历史仍主导 → 攻击者获得高权重 → Recall=0%

### 间接关系

1. **信任分数方差大** → **tau_t 分位数极端** → **EMA 累积极端值** → **阈值失效**
   - 失败案例 `trust_std` 均值: **15.85**
   - 成功案例 `trust_std` 均值: **0.30**
   - 方差大导致分位数极端，EMA 累积后阈值失效

2. **对齐分数为正** → **历史累积正对齐** → **高信任分数** → **高权重** → **Recall 失败**
   - 失败案例攻击者对齐分数: **+0.34**
   - 成功案例攻击者对齐分数: **-0.23**

## 改进建议

### 短期修复 (立即实施)

1. **对 tau_t 进行 clamp**:
   ```python
   tau_t = torch.clamp(tau_t, min=-1.0, max=0.0)
   ```
   - 防止阈值过度负向
   - 避免极端值累积

2. **降低 tau_ema_beta**:
   ```python
   tau_ema_beta = 0.8  # 或 0.85，而不是 0.95
   ```
   - 使阈值更敏感，能更快响应变化
   - 新值权重 20%，历史权重 80%

3. **降低 rho**:
   ```python
   rho = 0.7  # 或 0.8，而不是 0.9
   ```
   - 使当前信息更重要
   - 当前信息权重 30%，历史权重 70%

### 中期改进

1. **引入重置机制**:
   - 当检测到异常时（如信任分数方差过大），重置相关历史信息
   - 防止历史信息"污染"当前判断

2. **使用自适应衰减系数**:
   - 根据信任分数方差动态调整 `tau_ema_beta`
   - 方差大时降低 beta（更敏感），方差小时提高 beta（更稳定）

3. **分离不同历史信息**:
   - 对齐历史、范数历史、sketch 历史使用不同的衰减系数
   - 对齐历史可能需要更低的 rho（更敏感），范数历史可以保持较高的 rho（更稳定）

### 长期优化

1. **使用滑动窗口而非 EMA**:
   - 使用固定长度的滑动窗口存储历史信息
   - 避免 EMA 的累积效应

2. **引入异常检测**:
   - 检测到极端值时，不更新 EMA，而是使用异常值处理机制

## 总结

**衰减参数与 Recall 失败有直接关系**:

1. 🔴 **tau_t EMA (beta=0.95)**: 导致阈值被"锁定"在负值，无法有效区分攻击者
2. 🟠 **rho (0.9)**: 导致历史信息主导，掩盖当前异常

**最关键的参数**: `tau_t_ema` 的 EMA 累积效应是导致 Recall 失败的主要原因之一。

**建议优先修复**: 
- 对 `tau_t` 进行 clamp 限制
- 降低 `tau_ema_beta` 到 0.8
- 降低 `rho` 到 0.7-0.8
